# 스마트로페이 결제 연동 데이터 흐름

## 환경변수

| 값 | 설명 | 발급 주체 | 보관 위치 | 용도 |
|---|---|---|---|---|
| **Mid** | 가맹점 ID (10자리) | 스마트로 발급 | 프론트 + 백엔드 | 결제창 호출 시 가맹점 식별 (public) |
| **MerchantKey** | 상점키 (긴 문자열) | 스마트로 발급 | 백엔드만 | EncryptData/HashData 서명 생성 (private) |
| **CancelPwd** | 취소 비밀번호 | 관리자가 smilebiz에서 설정 | 백엔드만 | 취소 API 인증 |

---

## 전체 플로우

```
[Step 1]           [Step 2]            [Step 3]           [Step 4]            [Step 5]
백엔드             프론트               스마트로            프론트 success      백엔드
EncryptData 생성 → 결제창 호출 ──────→  결제 처리 ──────→  Tid/TrAuthKey ───→  승인 API 호출
                                                          수신                → DB 저장
```

---

## Step 1. EncryptData 생성 (백엔드)

위변조 방지용 암호화 데이터. **MerchantKey가 포함되므로 반드시 백엔드에서 생성.**

### IN

| 파라미터 | 설명 | 값 예시 | 출처 |
|---------|------|--------|------|
| EdiDate | 전문생성일시 (YYYYMMDDHHmmss, 14자리) | `20260225143000` | 서버 현재시간 |
| Mid | 가맹점 ID | `testMID123` | 환경변수 |
| Amt | 결제금액 (숫자만, 콤마 없음) | `50000` | 프론트에서 전달받은 최종 결제금액 |
| MerchantKey | 상점키 (비공개) | `abcdef1234...` | 환경변수 |

### OUT

```
EncryptData = Base64( SHA256( EdiDate + Mid + Amt + MerchantKey ) )
```

| 파라미터 | 설명 |
|---------|------|
| EdiDate | 생성 시 사용한 전문생성일시 (프론트에 함께 전달) |
| EncryptData | Base64 인코딩된 SHA256 해시값 |

---

## Step 2. 결제창 호출 (프론트)

`smartropay.payment()` SDK 호출로 결제 팝업 표시.

### SDK 로드 URL

| 환경 | PC | Mobile |
|------|-----|--------|
| 테스트 (STG) | `https://tpay.smartropay.co.kr/asset/js/SmartroPAY-1.0.min.js` | `https://tmpay.smartropay.co.kr/asset/js/SmartroPAY-1.0.min.js` |
| 운영 (REAL) | `https://pay.smartropay.co.kr/asset/js/SmartroPAY-1.0.min.js` | `https://mpay.smartropay.co.kr/asset/js/SmartroPAY-1.0.min.js` |

### IN (form 파라미터)

| 파라미터 | 설명 | 값 예시 | 출처 |
|---------|------|--------|------|
| PayMethod | 결제수단 코드 | `CARD` | 고정 또는 사용자 선택 |
| Mid | 가맹점 ID | `testMID123` | 환경변수 |
| Moid | 주문번호 (가맹점 주문 식별자) | `RS02250001234` | **rsrvNo** (예약번호) |
| Amt | 결제금액 | `50000` | finalPrice (쿠폰 적용 후) |
| GoodsName | 상품명 | `공항주차 서비스 (3일)` | 계산 |
| ReturnUrl | 결제 완료 후 결과를 수신할 URL | `.../payment/success?rsrvCode=xxx` | 프론트 URL |
| BuyerName | 구매자명 | `홍길동` | 회원정보 또는 비회원 입력 |
| BuyerTel | 구매자 연락처 (숫자만) | `01012345678` | 회원정보 또는 비회원 입력 |
| BuyerEmail | 구매자 이메일 | `user@email.com` | 회원정보 |
| EdiDate | 전문생성일시 | `20260225143000` | Step 1에서 수신 |
| EncryptData | 암호화 데이터 | `a3f2b1c...` | Step 1에서 수신 |
| StopUrl | 결제 중단 시 이동 URL (**Mobile 필수**) | `.../payment/fail` | 프론트 URL |

### OUT

스마트로 결제 팝업이 표시됨.

---

## Step 3. 결제 완료 → ReturnUrl 리다이렉트

사용자가 결제를 완료하면 스마트로가 ReturnUrl로 POST 전송.

### OUT (스마트로 → 프론트 success 페이지)

| 파라미터 | 설명 | 비고 |
|---------|------|------|
| **Tid** | 거래번호 (30자리, 스마트로 발급) | 승인/취소 시 필수 |
| **TrAuthKey** | 인증키 | **유효시간 10분**, 승인 API 호출에 사용 |
| ResultCode | 인증 결과코드 | 인증 단계의 결과 |
| ResultMsg | 인증 결과메시지 | |
| Moid | 주문번호 | rsrvNo와 동일 |
| Amt | 금액 | |
| PayMethod | 결제수단 | |

---

## Step 4. 백엔드 승인 API 호출

프론트 success 페이지에서 Tid, TrAuthKey를 백엔드로 전송 → 백엔드가 스마트로 승인 API 호출.

### 4-1. 프론트 → 백엔드

| 파라미터 | 설명 | 출처 |
|---------|------|------|
| tid | 거래번호 | Step 3에서 수신 |
| trAuthKey | 인증키 | Step 3에서 수신 |
| rsrvCode | 예약 코드 | URL query |
| amount | 결제금액 | URL query |
| mCouponCode | 적용 쿠폰 코드 (회원만, 선택) | sessionStorage 등 |

### 4-2. 백엔드 → 스마트로 승인 API

| 환경 | URL |
|------|-----|
| 테스트 | `https://tapproval.smartropay.co.kr/payment/approval/urlCallApproval.do` |
| 운영 | `https://approval.smartropay.co.kr/payment/approval/urlCallApproval.do` |

```
POST {승인API URL}
Content-Type: application/json

{
    "Tid": "거래번호",
    "TrAuthKey": "인증키"
}
```

### 4-3. 스마트로 승인 응답 → 백엔드

| 필드 | 설명 | DB 매핑 |
|------|------|---------|
| ResultCode | 결과코드 (3001=카드성공) | `t_std_pay.result_code` (신규) |
| ResultMsg | 결과메시지 | `t_std_pay.result_msg` (신규) |
| **AuthCode** | 승인번호 | `t_std_pay.auth_code` **(신규, 취소 시 필수)** |
| **AuthDate** | 승인일자 (YYMMDDHHMMSS) | `t_std_pay.auth_date` **(신규, 취소 시 필수)** |
| Tid | 거래번호 | `t_std_pay.t_id` (기존) |
| Amt | 결제금액 | `t_std_pay.pay_price` (기존) |
| PayMethod | 결제수단 (CARD/BANK 등) | `t_std_pay.pay_method` (신규) |
| CardQuota | 할부개월 (00=일시불) | `t_std_pay.card_quota` (신규) |
| SignValue | 위변조 검증값 | 검증 후 저장 안 함 |
| CardNum | 카드번호 (마스킹) | `t_std_pay.credit_no` (기존) |
| Moid | 주문번호 | rsrvNo로 추적 가능 (저장 불필요) |

### SignValue 검증 (위변조 방지)

```
검증값 = Base64( SHA256( Tid[0:10] + ResultCode + Tid[10:15] + MerchantKey + Tid[15:] ) )
```

백엔드에서 생성한 검증값과 응답의 SignValue가 일치하면 정상 거래.

---

## Step 5. DB 저장

승인 성공 시 백엔드에서 트랜잭션 처리.

```
1. fn_make_key('PAY_CODE') → payCode 채번
2. INSERT t_std_pay (payCode, t_id, auth_code, auth_date, pay_price, ...)
3. UPDATE t_std_reservation SET pay_code = #{payCode}, rsrv_type = 'CONFIRMED'
4. (회원+쿠폰) UPDATE t_std_member_coupon SET m_coupon_lock = #{rsrvCode}
```

### t_std_pay 저장 매핑

| 컬럼 | 값 출처 | 기존/신규 |
|------|---------|----------|
| pay_code | fn_make_key('PAY_CODE') | 기존 (PK) |
| pay_date | NOW(6) | 기존 |
| pay_price | 승인응답 Amt | 기존 |
| credit_no | 승인응답 CardNum (마스킹) | 기존 |
| t_id | 승인응답 Tid | 기존 |
| pay_type | 'PAID' 등 | 기존 |
| reg_uid | userUid 또는 guestCode | 기존 |
| **auth_code** | 승인응답 AuthCode | **신규** |
| **auth_date** | 승인응답 AuthDate | **신규** |
| result_code | 승인응답 ResultCode | 신규 (권장) |
| result_msg | 승인응답 ResultMsg | 신규 (권장) |
| pay_method | 승인응답 PayMethod | 신규 (권장) |
| card_quota | 승인응답 CardQuota | 신규 (권장) |

## 취소 플로우 (참고)

### 취소 API URL

| 환경 | URL |
|------|-----|
| 테스트 | `https://tapproval.smartropay.co.kr/payment/approval/cancel.do` |
| 운영 | `https://approval.smartropay.co.kr/payment/approval/cancel.do` |

### IN (백엔드 → 스마트로)

| 파라미터 | 설명 | 값 출처 | 필수 |
|---------|------|--------|------|
| Mid | 가맹점 ID | 환경변수 | O |
| Tid | 거래번호 | `t_std_pay.t_id` | O |
| CancelPwd | 취소 비밀번호 | 환경변수 (smilebiz에서 설정) | O |
| CancelAmt | 취소 금액 | 요청값 | O |
| CancelMsg | 취소 사유 | 요청값 | |
| CancelSeq | 취소 차수 (1부터 순차) | 계산 | O |
| PartialCancelCode | 0=전체취소, 1=부분취소 | 요청값 | O |
| HashData | 검증 데이터 | 생성 | O |

### HashData 생성 (취소 전용)

```
HashData = Base64( SHA256( Tid + MerchantKey + CancelAmt + PartialCancelCode ) )
```

### OUT (스마트로 → 백엔드)

| 필드 | 설명 |
|------|------|
| ResultCode | 결과코드 |
| ResultMsg | 결과메시지 |
| CancelDate | 취소일자 (YYYYMMDD) |
| CancelTime | 취소시간 (HHmmss) |
| CancelNum | 취소번호 |
| CancelAmt | 취소금액 |
| RemainAmt | 부분취소 후 남은 금액 |

## DB 스키마 변경 (필요 작업)

### t_std_pay 컬럼 추가

```sql
ALTER TABLE t_std_pay
  ADD COLUMN auth_code VARCHAR(50) NULL COMMENT '승인번호 (취소 시 필수)' AFTER t_id,
  ADD COLUMN auth_date VARCHAR(14) NULL COMMENT '승인일자 YYMMDDHHMMSS (취소 시 필수)' AFTER auth_code,
  ADD COLUMN result_code VARCHAR(10) NULL COMMENT '결과코드' AFTER auth_date,
  ADD COLUMN result_msg VARCHAR(500) NULL COMMENT '결과메시지' AFTER result_code,
  ADD COLUMN pay_method VARCHAR(20) NULL COMMENT '결제수단 (CARD/BANK 등)' AFTER result_msg,
  ADD COLUMN card_quota VARCHAR(10) NULL COMMENT '할부개월 (00=일시불)' AFTER pay_method;
```

### t_sys_numbering에 PAY_CODE 추가

```sql
INSERT INTO t_sys_numbering (num_code, num_desc, num_prefix, seq_length, use_year, use_month, use_day, stack_count, rand_length)
VALUES ('PAY_CODE', '결제 코드', 'PY', 4, 'F', 'Y', 'Y', 0, '3');
```
